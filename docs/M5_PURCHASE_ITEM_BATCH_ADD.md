# M5-A2 采购订单商品批量添加

## 范围

M5-A2 在采购订单详情页增加独立的“批量添加商品”入口。它与单件“添加商品”并列，服务于同一采购订单的商品明细维护，不改变订单实付、退款、成本分摊、库存、验货、销售或售后事实。

## 批量规则

- 每次最多添加 50 行。
- 每行代表一条独立的 `PurchaseOrderItem`，数量固定为 1；批量添加不把数量字段解释为多件。
- 相同商品名和 SKU 也按独立明细保存，不自动合并。
- 每行可填写商品名称、SKU、商品参考成交总额和备注。
- 参考成交总额沿用 `PurchaseOrderItem.referenceAmount` 语义：它是整条明细的参考成交总额，不是单件价格、订单实付、分摊成本或库存成本。
- SKU 由服务端复用 `normalizeSku` 标准化；客户端不能提交标准化结果或 owner 信息。

## API 与事务

`POST /api/purchases/[purchaseOrderId]/items/batch` 只接受严格的 `items` 数组，未知字段、数量字段、成本字段、库存状态字段和 owner/order 注入字段均拒绝。所有行先通过同一套 Decimal 与文本校验，再由 `PurchaseOrderService.addPurchaseItemsBatch` 在一个事务中执行锁定检查和 `createMany`。

任一行失败，整批回滚，不产生部分明细。服务端复用 M5-A1 的 owner 隔离和下游锁定守卫：成本分摊、验货、库存、采购售后或采购退款存在时拒绝批量添加。已付款本身不构成锁定。

## 页面交互

批量弹窗初始包含一行，支持新增空白行、复制当前行、删除行、复制第一行生成 1 至 50 行、取消和一次性保存全部。至少保留一行；保存失败时保留用户填写内容并显示错误，成功后重新加载订单明细。

## 非目标

本刀不修改 Prisma schema 或 migration，不创建库存，不改变订单金额、退款、日报、验货、销售、售后、平台寄送或 `SOLD` 状态，也不新增采购单创建流程。

## 验收

`pnpm verify:m5-purchase-items` 覆盖独立明细、数量固定为 1、重复商品不合并、中文往返、参考成交总额、全批回滚、严格字段拒绝、50 行上限、各类下游锁定、跨 owner 隔离、订单与库存边界以及页面批量入口。当前通过 98 项有意义检查。
