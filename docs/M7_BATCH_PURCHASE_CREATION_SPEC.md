# M7-B1 新建采购订单批量录入

## 范围

M7-B1 在 `/purchases/new` 提供两种相互独立的录入模式：

- 普通录入：默认模式，保留原有商品行、数量和附件行为。
- 批量录入：面向多件独立实物；每行固定创建一条数量为 `1` 的 `PurchaseOrderItem`。

批量模式一张采购单最多 50 行，至少保留 1 行。相同商品名和 SKU 的多行仍是独立采购明细，系统不得自动合并。

## 批量行字段

每一行可填写：商品名称、SKU/规格、商品参考成交总额、生产日期、保质期月数和到期日期。

- 参考成交总额为整条采购商品明细的参考值；它不是订单实付、运费、成本分摊、库存成本、退款或报表计算依据。
- 生产日期、保质期月数和到期日期复用 M7-A1 的校验与计算规则。
- 手工填写的到期日期优先；到期日期为空且生产日期和保质期月数齐全时，服务端按日历月计算。
- 金额使用最多两位小数的非负 Decimal 字符串；空值保持未填写。

批量录入支持新增空白行、复制当前行、按指定数量复制首行，以及删除未提交行。复制仅复制当前行的商品和保质期输入，不会创建数据库记录；新增空白行不继承其他行的数据。

## 提交与原子性

客户端只提交当前选中模式的数据。服务端严格区分：

- 普通模式使用 `items`。
- 批量模式使用 `entryMode: "BATCH"` 和 `batchItems`。

批量 DTO 不接受客户端 `quantity`、`ownerId` 或未知字段。服务端把每条批量行固定为 `quantity = 1`，并在同一个数据库 transaction 内创建采购单及全部采购商品明细。任一行校验失败、保质期冲突或订单创建失败时，订单和商品明细均不落库。

## 明确不做

M7-B1 不修改 Prisma schema 或 migration，不创建库存、验货、分摊、物流、退款、采购售后、销售或日报数据；不修改订单实付、运费、净采购实付、库存成本或 `SOLD` 规则。订单附件仍遵循既有创建成功后上传语义，批量行不新增逐行附件能力。

## 验证

`pnpm verify:m7-purchase-create-batch` 通过独立临时 HTTP 服务验证：普通模式回归、批量独立行、相同商品/SKU 不合并、50 行上限、严格 DTO、保质期校验、事务回滚、并发独立创建、无库存/验货/SOLD/日报副作用，以及精确夹具清理。

M7-A2 未开始；本规格不包含采购明细批量操作的后续改造。
